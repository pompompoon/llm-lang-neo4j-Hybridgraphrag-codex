## 回答品質の自動評価（精度指標）

RAG の弱点である**ハルシネーション（コンテキストにない情報の生成）を定量的に検知する**ため、すべてのチャット回答に対して精度指標を自動算出し、回答と併せて UI に表示する。

評価は RAGAS 等で標準的に用いられる観点（Faithfulness / Answer Relevancy / Context Relevancy）を参考に、**ルールベース指標**と **LLM-as-a-Judge** の2段構成で設計した。ルールベース指標は追加依存なしで全プロバイダ（mock 含む）で動作するため、LLM が使えない環境や CI でも品質の回帰検証ができる。

```
回答生成
  ↓
① ルールベース指標（rag_metrics.py / 全プロバイダで動作）
  ↓
② LLM-as-a-Judge（PROVIDER が mock 以外のとき併用）
  ↓
総合品質 = （①の加重平均 + ②の平均）÷ 2
  ↓
0.7 未満なら検索クエリを変えて自動リトライ（最大2回）
```

### ① ルールベース指標

| 指標 | 検知対象 | 算出方法 |
|---|---|---|
| 忠実性（faithfulness） | ハルシネーション | 回答を文単位に分割し、各文の内容語が検索コンテキストに含まれる割合を平均 |
| 回答関連性（answer relevancy） | 質問の取りこぼし | 質問の内容語のうち回答に含まれる割合 |
| コンテキスト関連性（context relevancy） | 検索精度 | 検索ヒットのうち質問の語彙と重なるものの割合 |
| 数値根拠率（grounded numbers） | 数値のハルシネーション | 回答中の数値がコンテキストに実在する割合（数値なしは 1.0） |

内容語の抽出は日本語向けに実装した。形態素解析器に依存せず、漢字連続・カタカナ語・英数語を正規表現で抽出し、漢字語は2文字単位（文字バイグラム）でも照合することで、活用や複合語による表記ゆれを吸収している。

4指標の加重平均を内部スコアとする（忠実性 45% / 回答関連性 25% / コンテキスト関連性 15% / 数値根拠率 15%）。ハルシネーション抑止を最重要とみなし、忠実性に最大の重みを置いた。

### ② LLM-as-a-Judge

回答生成と同じ LLM に質問・検索コンテキスト・回答を渡し、忠実性・関連性・完全性の3軸を 0.0〜1.0 で JSON 採点させる。出力のパースに失敗した場合はルールベース値のみで判定を続行するフォールバック設計とし、LLM 評価単独には依存しない（自己採点バイアスへの対策を兼ねる）。

### 表示例と読み方

```
総合品質: 0.88 | 忠実性: 0.75 | 回答関連性: 1.00 | コンテキスト関連性: 0.20 | 数値根拠率: 1.00
LLM評価: 1.00（忠実 1.00 / 関連 1.00 / 完全 1.00）| リトライ: 0回 | ヒット: 5件
```

- 忠実性 0.75 → 回答の内容語の 75% が検索コンテキストで裏付けられた（残り 25% は LLM がコンテキスト外から補った語）
- コンテキスト関連性 0.20 → 検索ヒット5件のうち質問に直結するのは1件のみ。回答は正しくても**検索側に改善余地がある**ことを可視化できる

検算: ルールベース = 0.75×0.45 + 1.00×0.25 + 0.20×0.15 + 1.00×0.15 = **0.77**、総合品質 = (0.77 + 1.00) ÷ 2 = **0.88**

### 品質ゲートとしての利用

総合品質は表示用であると同時に、LangGraph 自律ループの**品質ゲート**を兼ねる。0.7 未満の場合はクエリ分析ノードに戻り、検索キーワードを変えて自動リトライする（最大2回）。従来は LLM の自己採点1軸のみ（ローカル LLM では固定値）だったゲートを、多軸の客観指標に置き換えた。

### 一括評価 API

回帰テスト用に、複数質問をまとめて評価するエンドポイントを用意した。

```
POST /api/evaluate
Body: {"items": [{"question": "東京都とは", "expected": "関東地方にある日本の首都"}]}
```

質問ごとの各指標に加え、全体平均と、`expected`（期待回答）を指定した場合は期待回答との内容語 F1（`expected_match`）を返す。プロンプトや検索パラメータ変更時の品質劣化を数値で検出できる。
